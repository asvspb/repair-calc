# План улучшения системы сохранения данных

## Текущее состояние

Система сохранения состоит из:
- **Клиент:** `StorageManager` → `IStorageProvider` (LocalStorage / API)
- **Синхронизация:** `ApiStorageProvider` с очередью запросов и rate limiting
- **Сервер:** Express + MySQL (repositories: ProjectRepository, RoomRepository)
- **Управление состоянием:** `ProjectContext` с debounce-сохранением

---

## Выявленные проблемы

### 🔴 Критические (потеря данных / сломанная логика)

#### 1. Нарушение контракта IStorageProvider
**Файл:** `src/api/storage/apiStorageProvider.ts:245-258`
**Проблема:** Метод `set()` для ключа PROJECTS бросает исключение `StorageProviderError('Use saveProjectsAsync')`. Это нарушает интерфейс `IStorageProvider` — синхронный контракт `set<T>()` не должен использоваться как сигнал "вызови другой метод".
**Последствие:** `StorageManager.saveProjects()` ловит это исключение, но `ProjectContext.scheduleSave` всё равно вызывает `saveProjectsAsync` напрямую. Два пути сохранения конфликтуют.

#### 2. sync/push — заглушка
**Файл:** `server/src/routes/sync.ts:20-26`
**Проблема:** Endpoint `POST /api/sync/push` помечает все изменения как синхронизированные без реальной обработки. Комментарий: `TODO: Implement actual change processing with conflict detection`.
**Последствие:** Невозможна двусторонняя синхронизация. Клиент полагается только на `sync/pull` + CRUD операции.

#### 3. Нет транзакций при обновлении проект + комнаты
**Файл:** `server/src/db/repositories/project.repo.ts`, `room.repo.ts`
**Проблема:** Обновление проекта и его комнат происходит в отдельных запросах без транзакции. При сбое между запросами данные остаются в несогласованном состоянии.

#### 4. Потеря pending-сохранений при перезагрузке
**Файл:** `src/utils/saveQueue.ts` — очередь хранится в памяти.
**Файл:** `src/contexts/ProjectContext.tsx:608-625` — `beforeunload` сохраняет в localStorage, но серверные изменения теряются.
**Последствие:** Если страница закрыта во время async-сохранения на сервер, изменения утеряны.

#### 5. Проглатывание ошибок синхронизации комнат
**Файл:** `src/api/storage/apiStorageProvider.ts:479-486`
**Проблема:** Ошибки в `syncRooms` логируются, но не пробрасываются наверх. Пользователь не узнаёт об ошибке, данные комнат не сохраняются на сервере.

### 🟡 Серьёзные (архитектурные проблемы)

#### 6. Sync/Async несоответствие в интерфейсе
**Файл:** `src/types/storage.ts:5-37`
**Проблема:** `IStorageProvider` имеет синхронный API (`get/set`), но API-операции по природе асинхронны. Это приводит к обходным решениям: кэширование, выброс исключений из `set()`, параллельные async-методы.
**Решение:** Интерфейс должен поддерживать async-операции: `getAsync<T>()`, `setAsync<T>()`.

#### 7. Дублирование fetchJson
**Файлы:** `src/api/projects.ts:118-222`, `src/api/rooms.ts:99-153`
**Проблема:** Два независимых `fetchJson()`. В `projects.ts` есть обработка 401 с refresh token, в `rooms.ts` — нет. При истечении токена запросы к комнатам падают без попытки обновления.
**Решение:** Единый HTTP-клиент с interceptors.

#### 8. Отсутствие разрешения конфликтов версий
**Файлы:** `server/src/routes/projects.ts:72-75`, `src/api/storage/apiStorageProvider.ts:573`
**Проблема:** Сервер проверяет `version` (optimistic locking), но клиент НЕ отправляет version (закомментировано). При конфликте возвращается 403, но клиент не умеет мержить или показывать конфликт пользователю.

#### 9. Отсутствие инвалидации кэша между вкладками/устройствами
**Файл:** `src/api/storage/apiStorageProvider.ts:48` — `CACHE_TTL = 30000`
**Проблема:** Если данные изменены в другой вкладке или на другом устройстве, кэш не обновляется до истечения TTL. Нет механизма `BroadcastChannel` для уведомления.
**Решение:** `BroadcastChannel` для синхронизации между вкладками.

#### 10. Сложная миграция ID (local → server)
**Файл:** `src/utils/idMapper.ts`, `src/api/storage/apiStorageProvider.ts:284-460`
**Проблема:** ~200 строк кода для маппинга `local-*` ID → серверные UUID. Маппинги хранятся в localStorage, что ненадёжно. Логика определения "нужна ли миграция" хрупкая.
**Решение:** Генерировать UUID на клиенте с начала, убрать `local-*` ID полностью.

### 🟢 Качество кода / мелкие проблемы

#### 11. Отсутствие валидации данных при загрузке
**Файл:** `src/utils/storage.ts:76-78`
**Проблема:** Проверяется только `Array.isArray()`, но не структура элементов. Повреждённый проект без обязательных полей вызовет crash.

#### 12. Использование `require()` в ES-модуле
**Файл:** `src/api/storage/apiStorageProvider.ts:703`
**Проблема:** `const { LocalStorageProvider } = require(...)` — не работает в Vite/ESM. Потенциально ломает tree-shaking.

#### 13. Множественные пути сохранения
**Файлы:** `ProjectContext.tsx` — `scheduleSave`, `scheduleTotalsSave`, прямые вызовы `StorageManager.saveProjects()`
**Проблема:** Три механизма сохранения работают параллельно, нет координации. `deleteProject` вызывает `StorageManager.saveProjects()` напрямую, минуя debounce.

#### 14. Дублирование типов ApiRoom
**Файлы:** `src/api/projects.ts:30-51`, `src/api/rooms.ts:16-36`
**Проблема:** Интерфейс `ApiRoom` определён в двух файлах. Изменение в одном не обновит другой.

#### 15. Дублирование утилит parseJSON и apiToClientRoom
**Файлы:** `src/api/projects.ts:95-102` и `src/api/rooms.ts:64-71`
**Проблема:** Одинаковые функции в двух файлах.

## Сценарии использования

### Загрузка проекта через локальный JSON-бэкап

**Контекст:** Пользователь загружает JSON-файл с несколькими проектами (объектами/квартирами).

#### Текущее поведение (до улучшений)

```
1. Пользователь: BackupManager → "Загрузить бэкап (JSON)"
2. Чтение файла → StorageManager.importFromJSON()
3. Валидация: проверка структуры projects[], rooms[], workTemplates[]
4. Диалог подтверждения: "Найдено N проектов. Заменить текущие данные?"
5. При подтверждении:
   - onImport(projects, activeProjectId) → полная замена в localStorage
   - Если авторизован: ApiStorageProvider.saveProjectsAsync() → отправка на сервер
   - Ошибка: set() бросает исключение "Use saveProjectsAsync" → конфликт путей сохранения
6. Результат: данные в localStorage, но возможна потеря при синхронизации с сервером
```

**Проблемы текущей реализации:**
- Нет различия между "заменить всё" и "добавить к существующим"
- При синхронизации с сервером возможны конфликты ID (local-*)
- Ошибки синхронизации комнат глотаются — пользователь не видит, что данные не ушли на сервер
- Нет валидации структуры элементов (только Array.isArray)

#### Новое поведение (после Фазы 1)

```
1. Пользователь: BackupManager → "Загрузить бэкап (JSON)"
2. Чтение файла → StorageManager.importFromJSON()
3. Валидация: 
   - Zod-схемы для каждого проекта, комнаты, работы
   - Детальный отчёт об ошибках валидации
4. Диалог с выбором действия:
   ┌─────────────────────────────────────────────┐
   │ Загружено 3 проекта из бэкапа              │
   │                                             │
   │ ○ Заменить все текущие проекты             │
   │ ○ Добавить к существующим (слияние)        │
   │ ○ Показать список для выбора               │
   │                                             │
   │ [Отмена]              [Импортировать]      │
   └─────────────────────────────────────────────┘
5. При импорте:
   - Генерация UUID для новых проектов/комнат (если ID отсутствуют)
   - Единый путь сохранения: ProjectContext.scheduleSave()
   - Debounce + сохранение в localStorage
   - Async-сохранение на сервер через saveProjectsAsync()
   - Сохранение pending-очереди в localStorage
6. UI-индикатор:
   - "Сохранение..." → spinner
   - "Сохранено" → checkmark
   - "Ошибка синхронизации" → warning + retry button
7. При закрытии вкладки во время сохранения:
   - pending-очередь восстанавливается при следующем открытии
   - Автоматическая отправка отложенных запросов
```

#### После Фазы 2 (архитектурные улучшения)

```
Дополнительно:
- Единый HTTP-клиент с автоматическим refresh token
- При конфликте версий — диалог выбора:
  ┌─────────────────────────────────────────────┐
   │ Конфликт данных                            │
   │                                             │
   │ Проект "Квартира на Ленина" изменён на     │
   │ другом устройстве. Выберите действие:      │
   │                                             │
   │ ○ Перезаписать серверную версию            │
   │ ○ Сохранить как новую копию                │
   │ ○ Отменить, загрузить с сервера            │
   │                                             │
   │ [Отмена]              [Применить]         │
   └─────────────────────────────────────────────┘
- BroadcastChannel синхронизация между вкладками
- Упрощённая ID-логика без local-* маппинга
```
