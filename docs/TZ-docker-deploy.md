# Техническое задание
## Локальное развертывание проекта в Docker

### 1. Цель
Обеспечить возможность быстрого и воспроизводимого локального запуска проекта `repair-calc` в Docker-контейнере(ах) без необходимости ручной установки Node.js и других зависимостей на хост-машину.

Результат: разработчик должен уметь выполнить одну-две простые команды (например, `docker compose up`) и получить работающее приложение, доступное в браузере.

---

### 2. Описание текущего проекта

1. Стек:
   - **Фронтенд**: React 19 + TypeScript, Vite.
   - **Стили**: Tailwind CSS v4.
   - **Язык**: TypeScript, модульный формат ESM.
2. Запуск локально (без Docker):
   - Команда разработки: `npm run dev` (Vite, порт по умолчанию 3000).
   - Продакшн-сборка: `npm run build` → артефакты в каталоге `dist`.
   - Просмотр продакшн-сборки: `npm run preview`.
3. Дополнительные зависимости (на будущее):
   - `express`, `better-sqlite3`, `dotenv` — потенциально для backend/серверной части, но в текущей версии проекта не используются.

---

### 3. Общие требования к Docker-окружению

1. **Поддерживаемая платформа**: Docker Desktop (Windows/macOS) и Docker Engine (Linux).
2. **Минимальные версии**:
   - Docker Engine: 20.10+
   - Docker Compose: v2+
3. **Архитектура**: amd64 (x86_64). При возможности предусмотреть работоспособность на arm64 (Apple Silicon) подбором базового образа.
4. **Базовый образ**:
   - Рекомендуется использовать официальный образ Node.js LTS, например: `node:22-alpine` или актуальный LTS на момент реализации.

---

### 4. Варианты сценариев запуска

Реализо��ать два сценария:

#### 4.1. Сценарий A — Dev-режим (горячая перезагрузка)

Назначение: удобная разработка в контейнере.

Требования:
1. Запуск команды `npm install` и `npm run dev` выполняется внутри контейнера.
2. Хостовая директория проекта монтируется в контейнер (volume bind), чтобы изменения в коде были видны без пересборки образа.
3. Порт Vite (по умолчанию 3000) пробрасывается наружу.
4. HMR должен работать, если не установлен `DISABLE_HMR=true`.
5. Можно использовать отдельный `docker-compose.dev.yml` **или** параметры профилей в едином `docker-compose.yml`.
6. Достаточно одного сервиса `frontend`.

Рекомендуемый UX:
- Команда запуска dev:
  ```bash
  docker compose -f docker-compose.yml --profile dev up --build
  ```
  либо
  ```bash
  docker compose -f docker-compose.dev.yml up --build
  ```

#### 4.2. Сценарий B — Prod-режим (сборка и статика)

Назначени��: эмуляция продакшн-сборки.

Требования:
1. Многоэтапная сборка Docker-образа:
   - **build stage**:
     - Базовый образ: Node.js LTS.
     - Копирование `package.json`, `package-lock.json`/`pnpm-lock.yaml`/`yarn.lock` (если есть).
     - Установка зависимостей.
     - Копирование остального кода.
     - Выполнение `npm run build`.
   - **runtime stage**:
     - Использовать либо тот же Node.js образ, либо легкий nginx / static file server.
     - Копировать артефакты сборки из `dist`.
2. В runtime-стадии приложение должно обслуживаться на HTTP-порту 80 (или 4173/4174, если будет использоваться `vite preview`), но предпочтителен 80 для простоты.
3. Prod-контейнер не должен содержать dev-зависимости и исходники TS/JSX — только необходимый рантайм и артефакты сборки.
4. Команда запуска prod (пример):
   ```bash
   docker compose -f docker-compose.yml --profile prod up --build -d
   ```

### 5. Dockerfile: требования

Создать файл `Dockerfile` в корне проекта с поддержкой обоих сценариев.

#### 5.1. Общие требования

1. Использовать **многоэтапную сборку** (`builder`, `production`), чтобы итоговый образ был как можно меньше.
2. Обрабатывать переменные окружения:
   - `GEMINI_API_KEY` — прокидывается через аргумент/ENV при сборке или запуске, учитывая, что Vite читает его на этапе билда через `loadEnv`.
3. Конфигурация должна позволять:
   - Запуск в dev-режиме (через `npm run dev`).
   - Запуск в prod-режиме (через server, например nginx или node + `serve`).

#### 5.2. Примерная структура Dockerfile (описание для разработчика)

1. **builder** stage:
   - `FROM node:<lts>-alpine AS builder`
   - `WORKDIR /app`
   - Копировать `package*.json`.
   - Выполнить `npm install`.
   - Копировать остальной исходный код.
   - Выполнить `npm run build`.

2. **production** stage (есл�� используем nginx):
   - `FROM nginx:alpine AS production`
   - Копировать `dist` в `/usr/share/nginx/html`.
   - Настроить `nginx.conf` (при необходимости) для корректной отдачи SPA (fallback на `index.html`).

3. **dev** запуск возможен:
   - Либо на базе того же `node:<lts>-alpine` в отдельном `docker-compose` сервисе.
   - Либо через отдельный `Dockerfile.dev` (по усмотрению исполнителя, но предпочтителен один `Dockerfile`).

---

### 6. docker-compose: требования

Создать файл `docker-compose.yml` в корне проекта.

#### 6.1. Сервис `frontend-dev`

1. Использует образ на базе Node (можно билдить из локального Dockerfile или использовать официальный образ напрямую).
2. Рабочая директория `/app`.
3. Монтирование локальных файлов:
   - `./:/app`
   - Исключить node_modules (через volume для `/app/node_modules`) или оставить их внутри контейнера.
4. Команда по умолчанию:
   - `npm install` (однок��атно) и `npm run dev -- --host 0.0.0.0 --port 3000`.
5. Проброс порта:
   - `3000:3000`.
6. Переменные окружения (опционально):
   - `GEMINI_API_KEY` (если нужен для фронта).
   - `DISABLE_HMR` (для отключения HMR при необходимости).
7. Использовать профиль `dev`:
   ```yaml
   profiles: ["dev"]
   ```

#### 6.2. Сервис `frontend-prod`

1. Использует образ, собранный на основе мультистейдж `Dockerfile` (builder + nginx).
2. Порт:
   - `8080:80` (или любой стандартный внешний порт, по которому фронтенд будет доступен).
3. Профиль `prod`.
4. Команда по умолчанию — стандартный entrypoint nginx (дополнительные команды не требуются).

---

### 7. Работа с переменными окружения

1. Используем `.env` в корне проекта (по аналогии с `.env.example`).
2. Для Docker требуется предусмотреть:
   - Возможность передавать `GEMINI_API_KEY` через:
     - `.env` файл в корне (docker-compose: `env_file: .env`).
     - или через `environment:` в `docker-compose.yml`.
3. Важно: Vite читает переменные среды при сборке/старте (`loadEnv`). При изменении `.env` необходимо пересобирать образ (`docker compose build`).

### 8. Документация для разработчиков

Добавить в `README.md` раздел "Локальный запуск в Docker" со следующими пунктами:

1. Предварительные требования:
   - Установлены Docker и Docker Compose.
2. Запуск dev-режима:
   ```bash
   docker compose --profile dev up --build
   ```
   - Открыть `http://localhost:3000`.
3. Остановка dev-режима:
   ```bash
   docker compose --profile dev down
   ```
4. Запуск prod-режима:
   ```bash
   docker compose --profile prod up --build -d
   ```
   - Открыть `http://localhost:8080`.
5. Остановка prod-режима:
   ```bash
   docker compose --profile prod down
   ```

### 9. Нефункциональные требования

1. **Размер образа**:
   - Продакшн-образ должен быть по возможности минимальным (ис��ользовать Alpine-образы, многоэтапную сборку).
2. **Безопасность**:
   - Не хардкодить значения секретов (например, `GEMINI_API_KEY`) в Dockerfile или репозитории.
   - Использовать `.env` / переменные окружения.
3. **Производительность разработки**:
   - Монтаж исходников с хоста должен обеспечивать приемлемую скорость (Alpine-образ и bind mount на macOS/Windows могут работать медленнее — это нормально, но требуется избегать ненужных громоздких volume).

### 10. Критерии принятия

1. В репозитории присутствуют файлы:
   - `Dockerfile` в корне.
   - `docker-compose.yml` в корне.
   - Обновленный `README.md` с инструкциями по работе с Docker.
   - Данный файл технического задания (`TZ-docker-deploy.md`).
2. На чистой машине, где установлен только Docker и Docker Compose, после клонирования репозитория и наличия корректного `.env`:
   - Команда `docker compose --profile dev up --build` успешно поднимает dev-среду (Vite) и приложение доступно по `http://localhost:3000`.
   - Команда `docker compose --profile prod up --build -d` успешно собирает прод-образ и приложение доступно по `http://localhost:8080`.
3. Все описанные в README команды работают без ошибок при соблюдении условий (наличие Docker, свободные порты 3000 и 8080).
